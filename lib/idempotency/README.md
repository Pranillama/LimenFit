# lib/idempotency

Idempotency key helpers for API route handlers. Implemented in T7 (d8d2a50b-03fe-4c61-95d2-7f0af2a9f1f2).

## Purpose

Provides server-side deduplication for workout mutation API routes. Each queued
mutation carries a `clientMutationId` (generated by `buildWorkoutCreateMutation` et al.
in `features/workout/store/mutationDescriptors.ts`) that the client sends with every
request. The server uses the helpers here to detect and short-circuit duplicate retries.

## Files

| File | Boundary | Description |
|------|----------|-------------|
| `index.ts` | client + server | `newClientMutationId()` — generates UUID keys, imported by the Zustand store |
| `server.ts` | server only | `withIdempotency<T>()` — deduplicates via `mutation_receipts` |

## Server contract (`server.ts`)

### `withIdempotency<T>(opts)`

```ts
interface WithIdempotencyOptions<T> {
  supabase: SupabaseClient<Database>;
  userId: string;
  clientMutationId: string;  // must be a v4 UUID
  mutationType: string;       // e.g. 'workout.create'
  resourceType: string;       // e.g. 'workout'
  handler: () => Promise<{ resourceId: string | null; response: T }>;
}

interface IdempotencyResult<T> {
  replayed: boolean;
  resourceId: string | null;
  response: T | null;  // null on replay — caller reconstructs from resourceId
}
```

**Validation:** `clientMutationId` is validated as a v4 UUID before any DB work.
A malformed ID throws `IdempotencyValidationError`; callers map this to a 400.

**First call (cache miss):**

1. Looks up `mutation_receipts` by `(client_mutation_id, user_id)` — no row found.
2. Calls `handler()` to run the actual mutation.
3. Inserts a receipt row with `(client_mutation_id, user_id, mutation_type, resource_type, resource_id)`.
4. Returns `{ replayed: false, resourceId, response }`.

**Replay (cache hit):**

The pre-existing receipt is returned immediately without running `handler()`:
`{ replayed: true, resourceId: <existing>, response: null }`.

The caller is responsible for constructing the success response from `resourceId`:

- **Create mutations** (e.g. POST `/api/workouts`): re-fetch the existing row using
  `resourceId` to echo back full metadata to the client.
- **Update/delete mutations** (e.g. PATCH, DELETE): return a static success body —
  re-fetching is unnecessary because the operation is already reflected in the DB.

**Race condition (concurrent identical requests):**

If two requests pass the cache-miss check simultaneously, the second `INSERT` will hit a
unique-violation (`23505`). The wrapper catches this, re-reads the winning receipt, and
returns `{ replayed: true, resourceId, response: null }` — identical to a normal replay.

## RLS note

`mutation_receipts` has Row Level Security policies that restrict reads and writes to the
authenticated user. Passing the anon/session-scoped Supabase client (from
`createSupabaseServerClient()`) is sufficient — `userId` is used as an explicit filter
in addition to RLS as a defence-in-depth measure.

## Client-side contract (T7 pairing)

The store's mutation queue (`ActiveWorkoutState.queue`) holds `QueuedMutation` objects.
Each mutation has:

- `clientMutationId` — stable UUID per logical operation, generated at enqueue time.
- `attempts` — incremented by `store.incrementAttempt()` on each retry.
- `dependsOnLocalIds` — local IDs the server must have resolved before this mutation
  can apply (e.g. a `set.log` depends on its parent `workoutExercise.add`).

The flush engine (`features/workout/store/queue.ts`) sends `clientMutationId` in the
request body. This module's helpers on the server verify the ID has not been seen before,
apply the mutation, and store the ID so retries are no-ops.

## Hydration note

When `hydrateActiveWorkout` (`features/workout/store/hydration.ts`) restores an
`in_progress` workout from the server, the hydrated snapshot has an **empty queue** —
all server records already exist and no mutations need to be replayed. New user actions
after hydration generate fresh mutations with new `clientMutationId`s following the
normal T7 dedupe contract.
